# 범위 (Scope) — MVP

청약핏에서 **무엇을, 얼마나, 어떤 순서로** 만드는지를 관리한다. (제품 정의는 [README](../README.md), 코드 구조는 [architecture.md](./architecture.md), 범위 결정의 근거는 [decisions.md](./decisions.md))

## 0. 두 가지 분류 기준

한정된 리소스에서 **무엇을 먼저, 무엇을 꼭** 만들지 정하기 위해 기능을 두 가지 기준으로 분류한다.

- **기준 1 — 중요도 (MoSCoW)**: 이게 _얼마나_ 중요한가?
  - **Must** — 없으면 MVP가 성립 안 됨
  - **Should** — 중요하지만 없어도 출시 가능 (다음에 바로 채움)
  - **Could** — 있으면 좋은 것 (nice-to-have)
  - **Won't** — 이번엔 명시적으로 안 함 (범위 침범 방지)
- **기준 2 — Phase (순서)**: 이걸 _언제(몇 번째로)_ 만드나?
  - **Phase 1 뼈대** → **Phase 2 핵심 가치(진짜 MVP)** → **Phase 3 완성도·편의**

즉 모든 기능은 (중요도, Phase) **두 라벨**을 가진다.

> 제품은 **[A] 자격 판정**(넣을 수 있나) 하나에 집중한다. 한때 검토했던 **당첨 시 자금 계획(자기자본·대출)은 범위 밖**으로 제외했다. (멘토링 Breadth>Depth — [decisions.md](./decisions.md) D15)

## 1. Phase 정의

| Phase                              | 한 줄 정의                                                                |
| ---------------------------------- | ------------------------------------------------------------------------- |
| **Phase 1 — 뼈대**                 | 로그인 + 프로필 + PDF 업로드(저장만)                                      |
| **Phase 2 — 핵심 가치 (진짜 MVP)** | 파싱 → 요약 제시 → **[A] 자격 판정(근거+서류)** → 결과                    |
| **Phase 3 — 완성도·편의**          | 어드민(룰 편집), 가점 계산기, 공고 요약 카드, UX 개선, 분석 이력, 주의사항 강조 |

## 2. 기능 목록 (Phase별)

> 기능을 **Phase별 표**로 나누고, 각 기능에 **중요도(MoSCoW)** 를 열로 적는다.

### Phase 1 — 뼈대

| 기능                                                 | 중요도 | 비고                                                                                                                                                                                                |
| ---------------------------------------------------- | ------ | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **룰·필드 사전 정의 (법령·공고 조사)**               | Must   | 🧱 **선행 조건**. 코드 아닌 스펙 산출물. 자격(eligibility) 룰 6종 + 기준표(소득/가점/예치금) + 역산한 프로필·공고 필드. 산출물·방법·완료기준은 [domain/README.md](./domain/README.md) 채우는 법 |
| DB 스키마 / 마이그레이션 (profiles·notices·analyses) | Must   | 모든 기능의 토대                                                                                                                                                                                    |
| zod 스키마 정의 (프로필 검증 + 공고 추출 _모양_)     | Must   | 데이터 정의. 스키마를 _채우는_ 파싱은 P2                                                                                                                                                            |
| 카카오 로그인 (Supabase Auth)                        | Must   |                                                                                                                                                                                                     |
| 개인정보 접근 통제 (Supabase RLS)                    | Must   | 🔒 프로필=민감정보(소득 등). 본인 데이터만 접근                                                                                                                                                     |
| 프로필 페이지 — 입력·열람·수정                       | Must   | 전용 라우트(`/profile`) + 내비 진입점. **자격용 필드(소득 포함)**. 필드는 사전 정의에서 확정                                                                                                         |
| 공고 PDF 업로드 → Storage 저장                       | Must   |                                                                                                                                                                                                     |

### Phase 2 — 핵심 가치 (진짜 MVP)

| 기능                                                        | 중요도 | 비고                                                                                                 |
| ----------------------------------------------------------- | ------ | ---------------------------------------------------------------------------------------------------- |
| PDF 파싱 파이프라인 (텍스트추출 → OpenAI → zod)             | Must   | [A] 입력 공고 변수(규제지역·분상제·청약 가능지역·전용면적·공급유형/세대수·분양가 등) 추출. 한글 표/이미지 텍스트추출이 난제                          |
| 공고 추출결과 제시 (요약 겸, 선택적 수정)                   | Must   | 사용자가 PDF를 열람할 일 없음. 원하면 직접 수정만 가능                                               |
| 프로필 누락 항목 재확인 요청                                | Must   | 매칭에 필요한 프로필 값이 비면 "확인필요" + 되묻기                                                  |
| 공통기준 룰 DB 테이블 + 시드 (소득/가점/예치금)            | Must   | docs/domain 값을 **rule 테이블+시드 마이그레이션**으로 적재(코드 하드코딩 X). 엔진이 DB에서 로드 ([decisions.md](./decisions.md) D16) |
| **[A] 매칭 엔진 — 판정(가능/불가능/확인필요)**              | Must   | 순수 함수, TDD. 청약 6종 전부. 룰은 DB에서 로드해 인자 주입                                          |
| **[A] 매칭 엔진 — 판정 근거 표시**                          | Must   |                                                                                                      |
| **[A] 필요 서류 안내**                                      | Must   | 룰마다 requiredDocuments[] — 판정 근거와 1:1                                                         |
| 결과 화면 ([A] 자격 판정)                                   | Must   | 공고 요약 + 청약유형별 판정·근거·서류                                                                |
| 면책 고지 (결과는 참고용, 최종은 공고·청약홈 확인)          | Must   | 자격 판정도 참고용 — 신뢰 원칙                                                                       |
| 파싱 완전 실패 시 처리 (재시도 / 분석 불가 안내)            | Should | 실패해도 사용자에게 PDF 전사를 요구하지 않음                                                         |

### Phase 3 — 완성도·편의

| 기능                             | 중요도 | 비고                                                                 |
| -------------------------------- | ------ | -------------------------------------------------------------------- |
| 어드민 대시보드 (룰 값 편집)     | Should | DB rule 테이블 값을 코드 없이 갱신 ([decisions.md](./decisions.md) D17) |
| 일반공급 가점 계산기 (84점)      | Should |                                                                      |
| 공고 핵심 요약 카드              | Should | 검증된 구조화 데이터 렌더 (LLM 내러티브 X)                           |
| 분석 이력 저장/재활용 (analyses) | Should |                                                                      |
| 프로필 페이지 UX 개선            | Could  | P1의 기능형 폼을 사용성·미관 다듬기                                  |
| 파싱 중 로딩 UI                  | Could  | 파싱 대기 동안 보기 좋은 로딩/진행 표시                              |
| 공고 PDF 웹 검색·자동 수집       | Could  | 단지명 등으로 웹에서 공고 PDF를 찾아 업로드 대체. 수동 업로드는 유지 |

### Won't — 이번엔 안 함 (범위 밖)

| 기능                    | 비고          |
| ----------------------- | ------------- |
| 당첨 시 자금 계획 — 자기자본·대출(LTV/DSR)·납부 스케줄 | 한때 MVP에 포함했다 제외. 멘토링 Breadth>Depth — [A] 자격 판정에 집중 ([decisions.md](./decisions.md) D15). 필요 시 별도 프로젝트 |
| 공공주택(LH/SH) 룰 엔진 | 향후 확장     |
| 기관추천 특공 자동판정  | 자동판정 불가 |
| LLM 내러티브 요약       |               |
| 분석 비교/알림          |               |
| 청약 당첨 확률·경쟁률 예측 | 데이터·모델 필요, 불확실성 큼 — 범위 밖 |

## 3. Phase별 완료 정의 (Definition of Done)

> 각 항목은 "했다/안 했다"로 판정 가능해야 한다.

### Phase 1 — 뼈대

- [ ] **(선행) 룰·필드 사전 정의**가 완성된다 — 아래 항목들보다 먼저. (방법·완료기준: [domain/README.md](./domain/README.md) 채우는 법)
- [ ] profiles·notices·analyses DB 스키마가 마이그레이션으로 생성된다.
- [ ] 프로필 검증 / 공고 추출(모양) zod 스키마가 정의된다.
- [ ] 카카오 OAuth로 로그인/로그아웃이 동작한다.
- [ ] RLS가 걸려 다른 유저의 프로필·공고에 접근할 수 없다(검증 포함).
- [ ] 로그인한 유저가 프로필(자격용 필드, 소득 포함)을 전부 입력·저장한다.
- [ ] 프로필 페이지(`/profile`)에 내비로 진입해 저장된 값을 열람하고, 수정·재저장할 수 있다.
- [ ] 공고 PDF를 업로드하면 Supabase Storage에 저장되고, 내 공고 목록에 보인다.

### Phase 2 — 핵심 가치 (진짜 MVP) ← 출시 기준

- [ ] 공통기준 룰(소득/가점/예치금)이 **DB rule 테이블 + 시드 마이그레이션**으로 적재되어 엔진이 DB에서 읽는다(코드 하드코딩 X).
- [ ] 업로드한 PDF가 파싱되어 [A] 입력 공고 변수(규제지역·분상제·청약 가능지역·전용면적·공급유형 등)가 구조화 저장된다(zod 검증 통과).
- [ ] 파싱 결과가 요약 카드로 제시된다 — 사용자가 검수를 위해 PDF를 열람할 일이 없다(원하면 선택적 수정만 가능).
- [ ] **[A]** 매칭 엔진이 공고 내 모든 민영 청약유형(6종)에 대해 가능/불가능/확인필요를 판정한다.
- [ ] **[A]** 각 판정에 **근거**와 **필요 서류**가 함께 표시된다.
- [ ] 매칭에 필요한 프로필 값이 비면 단정하지 않고 "확인필요"로 처리하고, 무엇을 채울지 되묻는다.
- [ ] 결과 화면에 면책 고지가 표시된다.
- [ ] 매칭 엔진 단위 테스트(프로필 fixture × 공고 fixture)가 통과한다.
- [ ] 대표 공고 1건을 실제로 끝까지(업로드→자격 판정) 돌려본 골든 테스트가 통과한다.

### Phase 3 — 완성도·편의

- [ ] 어드민 대시보드에서 DB rule 테이블 값을 코드 배포 없이 편집·반영한다.
- [ ] 일반공급 결과에 84점 만점 가점이 항목별(무주택/부양가족/통장)로 분해 표시된다.
- [ ] 공고 핵심 요약 카드(일정·분양가·세대수·주의사항)가 검증된 구조화 데이터로 렌더된다.
- [ ] 프로필 페이지의 사용성·미관이 다듬어진다.
- [ ] 파싱 대기 중 보기 좋은 로딩/진행 UI가 표시된다.
- [ ] 분석 결과가 analyses에 스냅샷으로 저장되고 "내 분석 이력"에서 다시 본다.
- [ ] 전매제한·실거주의무·재당첨 제한 등 주의사항이 눈에 띄게 강조된다.
- [ ] 필드별 파싱 신뢰도가 결과/제시 화면에 시각적으로 표시된다(참고용).
- [ ] 단지명 등으로 공고 PDF를 웹에서 검색해 가져올 수 있다(수동 업로드 대체 경로).
