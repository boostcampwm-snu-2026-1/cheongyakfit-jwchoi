# 아키텍처 (Architecture)

청약핏이 **코드로 어떻게 돌아가는지**를 정의한다. (화면이 어떻게 보이는지는 [design.md](./design.md), 무엇을 언제 만드는지는 [scope.md](./scope.md), 청약 도메인 규칙은 [docs/domain/](./domain/README.md))

> 📌 이 프로젝트는 **자격 판정** 하나에 집중한다. 당첨 시 자금 계획(자기자본·대출)은 **범위 밖**으로 제외. ([decisions.md](./decisions.md) D15)

---

## 1. 핵심 설계 원칙

1. **파싱(LLM) ↔ CORE 엔진(순수)의 완전 분리**
   - 파싱 엔진: PDF → 구조화 데이터 (LLM 사용, 비결정적)
   - CORE 엔진(매칭): 구조화 데이터 + 프로필 → 자격 판정 (LLM **미사용**, 순수 함수, 결정론적)
   - 순수 함수이므로 단위 테스트가 쉽다. 이 서비스의 신뢰는 판정 정확도에서 나오므로 여기에 LLM 비결정성을 들이지 않는다.

2. **공통 기준은 DB 룰로 관리, 공고 변수만 추출**
   - 법령 공통기준(소득기준표·가점표·예치금표·신혼 7년 등)은 PDF에서 뽑지 않는다. 또한 자격 요건·법령은 생각보다 자주 바뀌므로 **코드에 하드코딩하지 않고 DB rule 테이블로 관리**한다 — 엔진은 런타임에 DB에서 로드해 인자로 주입받는다(순수성 유지). 값의 갱신은 어드민/마이그레이션으로(코드 배포 불필요). ([decisions.md](./decisions.md) D16)
   - "공통 기준"도 가구원수에 따라 값이 달라지고 매년 갱신되므로 단순 상수가 아닌 **기준표(룰)** 형태로 관리한다. 룰의 구조·근거·출처 SSoT는 [docs/domain/](./domain/README.md).
   - PDF에서는 **공고마다 달라지는 변수**만 추출한다.

3. **신뢰 우선 — 단, 사용자가 PDF를 다시 읽게 하지 않는다**
   - PDF 파싱은 100% 정확하지 않지만, 오류 검증 부담을 사용자에게 떠넘기지 않는다. 파싱 결과는 zod로 구조 검증하고 **깔끔한 요약 카드로 제시**한다(이 화면이 곧 "공고 핵심 요약" 가치). 사용자가 파싱 검수를 위해 원본 PDF를 열람하는 일은 **없다**(파싱 실패 시에도 PDF 전사를 요구하지 않음). 원한다면 추출값을 직접 수정할 수 있지만 선택사항이다.
   - 재확인이 필요한 건 **사용자가 직접 제공하는 정보(프로필)** 다. 매칭에 필요한 프로필 값이 비었거나 미흡하면 단정하지 않고 **"확인필요"**로 처리하고, 무엇을 채워야 하는지 사용자에게 되묻는다.
   - 공고 요약은 LLM 내러티브가 아니라 **검증된 구조화 사실을 그대로 렌더**한다.

## 2. 기술 스택

| 영역            | 선택                                           | 비고                                               |
| --------------- | ---------------------------------------------- | -------------------------------------------------- |
| 프레임워크/배포 | **Next.js (App Router) + TypeScript** / Vercel | 풀스택 단일 앱                                     |
| DB              | **Supabase Postgres**                          | NoSQL 아님 — [decisions.md](./decisions.md) 참조   |
| Auth            | **Supabase Auth (카카오 OAuth 단일 로그인)**   |                                                    |
| Storage         | **Supabase Storage**                           | 공고 PDF 원본                                      |
| LLM             | **OpenAI API**                                 | PDF 1차 파싱, 서버에서만 호출                      |
| 검증            | **zod**                                        | LLM 추출 결과 + 폼 검증 (structured output)        |
| 테스트          | **Vitest**                                     | 매칭 엔진은 순수 함수라 프레임워크 독립            |
| UI              | **Tailwind + shadcn/ui**                       | 토이에 빠르게 그릴 수 있는 조합                    |
| DB 접근         | **supabase-js + 생성 타입**                    | Prisma 안 씀 — [decisions.md](./decisions.md) 참조 |

## 3. 3층 구조

```
┌─────────────────────────────────────────────┐
│  WEB (Next.js App Router)                   │  ← 화면 + 라우팅
│  로그인·프로필페이지·업로드·요약·결과 화면            │
├─────────────────────────────────────────────┤
│  SERVER (Route Handlers / Server Actions)   │  ← 부수효과 담당
│  파싱(OpenAI 호출)·DB·Storage·Auth            │
├─────────────────────────────────────────────┤
│  CORE (순수 TS, 프레임워크 무관)                 │  ← 두뇌, 부수효과 없음
│  매칭 엔진 + 룰/기준표 + 도메인 타입               │
└─────────────────────────────────────────────┘
```

**제일 중요한 경계**: CORE는 Next.js·Supabase·OpenAI를 **전혀 import하지 않는다.** 순수 함수와 데이터뿐이라 단독 테스트가 쉽고, §1의 분리 원칙이 구조로 보장된다.

- 파싱(LLM, 비결정적) → SERVER
- 판정(순수, 결정적) → CORE

## 4. 폴더 레이아웃

```
app/              # Next.js 라우트 (화면들)
lib/
  core/           # ★ 순수 영역 (Vitest 대상)
    matching/     #   evaluate(profile, notice, rules) 청약유형별 자격 판정
    rules/        #   룰 타입 (값은 DB rule 테이블에서 로드해 주입). 구조·근거는 docs/domain/
    types/        #   도메인 타입
  server/         # 부수효과
    parsing/      #   PDF 텍스트추출 → OpenAI → zod
    db/           #   supabase 쿼리
    storage/      #   PDF 업로드
    auth/         #   카카오 OAuth
  schemas/        # zod (프로필 검증 + 공고 추출 §8)
supabase/         # 마이그레이션 + seed (rule 테이블 값 시드 포함 — D16)
```

## 5. 데이터 플로우

```
[카카오 로그인]  Supabase Auth
      │
[프로필 입력/수정] ──저장──► Supabase: profiles   (전용 페이지 /profile)
      │
[공고 PDF 업로드] ──► Supabase Storage
      │
[파싱 파이프라인]  PDF→텍스트추출 → OpenAI 구조화 → zod 검증
      │                                   └─► Supabase: notices (구조화 저장)
      ▼
[공고 요약 카드로 제시]  (검수 강제 없음 · 원하면 선택적 수정)
      │
[CORE 순수 함수]  프로필 × (룰[DB에서 로드해 주입] + 공고변수)
      │   └─ 매칭 엔진 → 청약유형별 판정·근거·서류
      │   (필요한 프로필 값이 비면 "확인필요" + 사용자에게 되묻기)
      ▼
[결과 화면]  ① 공고 핵심 요약  ② 청약유형별 판정·근거·서류·가점
```

## 6. 데이터 모델 (Supabase)

- **profiles** — 유저별 1개. 사용자 정보 스키마(§7). 자격용 필드(소득 포함).
- **notices** — 공고 1건당 1개(업로드, 추후 웹 수집도 가능 — scope P3). 구조화된 공고 변수(§8: 규제지역·분상제·청약 가능지역·분양가·세대수 등) + 원본 PDF 경로 + 필드별 파싱 신뢰도.
- **rule 테이블** — 기준표 **값**(소득/가점/예치금)의 런타임 SSoT(D16). 엔진이 로드해 인자로 주입, 어드민(D17)/마이그레이션으로 갱신. 값의 구조·근거·출처는 [docs/domain/](./domain/README.md).
- **analyses** — (profile × notice) 분석 결과 스냅샷 — 자격 판정 결과. "내 분석 이력"으로 재활용.

> 공고의 가변 구조(면적타입별·공급유형별 배열 등)는 Postgres **JSONB** 컬럼으로 둔다. 정형 필드는 컬럼, 가변 필드는 JSONB의 하이브리드.

### 개인정보 접근 통제 (RLS)

프로필은 생년월일·소득 등 민감정보다. Supabase **RLS(Row Level Security)** 로 본인 데이터만 접근하게 강제한다. (Prisma를 안 쓰는 결정적 이유 — [decisions.md](./decisions.md))

## 8. 공고 추출 스키마 (notices — PDF에서 뽑는 "변수")

> 🚧 **미정** — 추출할 정확한 필드·타입·enum은 Phase 1의 룰/필드 사전 정의에서 확정한다. ([domain/README.md 채우는 법](./domain/README.md))

> 주요 추출 대상: 분양가상한제·규제지역 여부, 청약 가능 지역, 평형(전용면적)·공급유형/세대수, 전매제한·실거주의무, 일정(접수·발표·계약), 평형별 분양가·세대수(요약 표시용).

## 9. CORE 엔진 (매칭)

CORE에는 순수 함수 매칭 엔진이 산다. `rules`(기준표)를 **인자로 주입**받아 순수성을 유지한다 — 룰 **값**은 SERVER가 **DB rule 테이블에서 로드**해 넘기고(D16), 룰의 구조·근거 SSoT는 [docs/domain/](./domain/README.md). 필요한 입력값이 없으면 단정하지 않고 **확인필요** 처리(허위 단정 금지).

### 매칭 엔진 — 자격 판정
- 청약유형마다 규칙 함수 하나:
  `evaluate(profile, notice, rules) → { status: 가능 | 불가능 | 확인필요, reasons[], requiredDocuments[] }`

> 🚧 **미정** — 유형별 판정 로직·가점 점수표·읽는 필드는 **Phase 1 룰/필드 사전 정의**에서 확정([docs/domain/eligibility.md](./domain/eligibility.md)).

## 10. 에러 / 신뢰도 처리

- 공고 파싱 결과는 zod로 구조 검증 후 **요약 카드로 제시**한다. 사용자가 검수를 위해 PDF를 열람할 일은 **없다**. 원하면 추출값을 직접 수정할 수 있으나 선택사항.
- zod 검증 실패/파싱 완전 실패 시 **재시도하거나 "분석 불가"를 안내**한다 — 사용자에게 PDF를 보고 값을 전사하라고 요구하지 않는다.
- **재확인은 프로필 쪽에 집중**: 매칭에 필요한 프로필 값이 비었거나 미흡하면 "확인필요"로 표기하고, 무엇을 채워야 하는지 사용자에게 되묻는다.

## 11. 테스트 전략

- **매칭 엔진(순수 함수)**: TDD. (프로필 fixture × 공고 fixture) → 기대 판정 단위 테스트.
- **파싱 파이프라인**: 샘플 공고 PDF 골든 테스트(대표 공고 몇 건의 추출 결과 스냅샷 검증).
