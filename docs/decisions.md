# 결정 기록 (Decisions)

"왜 그렇게 정했는지"를 남긴다. 결과 문서(scope·architecture 등)엔 결과만 적히므로, 이미 검토·기각한 선택을 다시 논의하지 않도록 여기에 근거를 모은다. (append-only, 지우지 않음. 결정이 바뀌면 새 항목으로 supersede)

> 분량이 커지면 `docs/adr/NNNN-title.md` 형식으로 승격한다. (지금은 단일 파일)

---

### D1. 범위는 중요도(MoSCoW) + Phase 두 라벨로 관리
모든 기능에 (중요도, Phase) 두 라벨 부여 — 행렬이 아니라 라벨이다. 마일스톤 명칭은 "M0/M1"이 아닌 **Phase 1~N** 사용(직관적, "0부터" 설명 불필요). 상세는 [scope.md](./scope.md).

### D2. 청약 6종 전부 P2(MVP)에 포함
일반공급 + 특공 5종(신혼·생애최초·다자녀·노부모·신생아). "일반공급+신혼부부만 먼저"도 검토했으나 6종 전부로 결정. 기관추천 특공은 자동판정 불가라 제외(안내만).

### D3. 공고 요약 카드는 P3 (P2 아님)
P2의 청약유형별 판정만으로 단독 출시 가능한 제품이 성립함. 요약은 가치를 더하지만 핵심 가치의 전제는 아님.

### D4. 프로젝트 구조 = 단일 Next.js 앱 + 폴더 분리 (모노레포 기각)
`lib/core`(순수)·`lib/server`(부수효과)·`app`(화면) 3층. 모노레포(packages 분리)는 경계를 물리적으로 강제하지만 토이엔 부대비용 과함. "폴더 + 순수 함수 규율"로 분리 원칙의 이점 대부분 확보.

### D5. DB = Supabase Postgres (NoSQL 기각)
데이터가 정형·관계형(profiles↔analyses↔notices)이고 RLS가 필요. NoSQL이 유리한 조건(비정형/초대규모/문서단위 읽기) 하나도 안 걸림. 공고의 가변 구조는 Postgres **JSONB**로 해결 → NoSQL로 갈 이유 소멸.

### D6. ORM 안 씀 — supabase-js 직접 사용 (Prisma 기각)
Prisma는 관리자 커넥션으로 붙어 **RLS를 우회**함 → P1 MUST인 "본인 데이터만 접근"이 무력화됨. 또 Prisma 마이그레이션과 Supabase 마이그레이션이 겹쳐 관리 포인트가 둘로 늘어남. supabase-js + 생성 타입으로 충분.

### D7. 개인정보 접근 통제(RLS)는 P1 MUST
프로필=민감정보(생년월일·소득). RLS 없으면 타인 데이터 노출 위험. 토이라도 보안 필수.

### D8. 기준표 SSoT = 문서(docs/domain/)
소득/가점/예치금/LTV/DSR 등 기준표의 진실원천은 [docs/domain/](./domain/README.md). 코드(`lib/core/rules/`)는 문서를 반영하며 역참조 주석 필수. 문서 변경 시 같은 PR에서 코드 동기화. (코드 SSoT 방식도 검토했으나 문서 SSoT로 결정)

### D9. 문서 구조 — 평평하게, 도구 누수 제거
- `docs/superpowers/`·`specs/` 폴더 제거(도구 이름이 경로에 새는 것 방지).
- 에이전트 컨텍스트 파일은 **CLAUDE.md만** 관리(AGENTS.md 표준은 다도구 환경용이라 단일 도구엔 불필요).
- Diátaxis 분류 폴더는 미리 만들지 않음(토이는 평평하게 시작, 커지면 분할).
- 초기 통합 설계문서(구 `*-design.md`)는 내용을 architecture·scope·domain으로 흩고 삭제(git 히스토리에 보존).
- `design.md`는 **UI/UX 디자인** 문서로 사용.
- **도메인 문서는 내용 확장(자격+자금)으로 `docs/domain/`로 분할** — `README`(인덱스·SSoT·채우는 법) / `overview`(배경·구분·용어) / `eligibility`(자격) / `finance`(자금). 제품 두 기둥(A/B)에 정렬. (D13)

### D10. 사용자가 PDF를 열람할 일은 없다 — 재확인은 프로필에 집중
"파싱이 부정확하니 추출값을 사용자가 교정한다"는 원래 방침은, 사용자가 원본 PDF를 다시 읽고 대조해야 한다는 뜻이 되어 **"공고를 안 읽어도 되게 해준다"는 서비스 목적에 정면으로 반함**.
- 공고 파싱 결과는 zod 검증 후 **요약 카드로 제시**만 한다(이 화면이 곧 "핵심 요약" 가치). 사용자가 검수를 위해 PDF를 열람하는 상황은 **없다(파싱 실패 포함)**. 파싱이 완전히 실패하면 재시도하거나 "분석 불가"를 안내할 뿐, PDF 전사를 요구하지 않는다.
- 원한다면 사용자가 추출값을 **직접 수정**할 수 있으나 이는 선택사항이지 요구되는 흐름이 아니다.
- **재확인은 사용자가 직접 제공하는 정보(프로필)에 집중**: 매칭에 필요한 프로필 값이 비거나 미흡하면 "확인필요"로 처리하고 무엇을 채울지 되묻는다. (소득 등 프로필 보완은 PDF와 무관하므로 정당)

### D11. 필요 서류 안내 = P2 Must
판정만 해주는 것과 "무엇을 준비해야 하는지"까지 알려주는 것은 사용자 체감 가치가 다름. 룰마다 `requiredDocuments[]`를 두면 판정 근거와 1:1 대응이라 추가 비용이 거의 없음 → P2 Must.

### D13. 제품 범위 확장 — "자격 판정" + "당첨 시 자금 계획" (둘 다 MVP)
자격 판정만으로는 가치가 절반. 당첨 시 **언제 얼마를 내야 하고(납부 스케줄) 대출이 얼마 나오는지(대출 가능액)** 가 실제 의사결정에 더 직접적. 두 기둥을 모두 P2(MVP)에 포함.
- **파싱 대상의 전환**: 소득요건·자격기준 같은 값은 **법령 고정**이라 파싱하지 않는다(코드 내장 — domain/overview.md §2). 대신 공고에서 뽑는 변수는 **평형별 분양가·계약금/중도금/잔금 일정·금액, 분양가상한제·규제지역 여부, 전매제한, 청약 가능지역** 등.
- **대출 계산은 DSR까지** 한다(LTV ∩ DSR ∩ 중도금대출 규정의 최솟값). → 프로필에 **재무 항목**(연소득은 기존, 기존 대출 연원리금 등) 추가 필요. 대출 금리·만기는 **가정값**(사용자 조정 가능) — 구체값 Phase 1 확정.
- 자금 계산기도 **순수 함수**로 CORE에 둔다(매칭 엔진과 동일 원칙). 돈 관련 추정이므로 면책 고지 필수.

### D12. 프로필 폼 전부 P1
프로필 필드는 전부 매칭 엔진의 입력이라 "매칭 필요분만 먼저" 같은 분리가 의미 없음. 폼 전체를 P1에 둠. 재방문 시 열람·수정이 필요하므로 전용 프로필 페이지(`/profile`)로 구현.
